# BAEM «Сильнейший» — состав проекта и честный статус

Цель: максимально сильный практический игрок = быть близким к GTO (низкая
эксплуатируемость) + доедать любого, кто отклоняется. Против идеального GTO
выиграть нельзя по определению; против любого реального оппонента и несовершенного
солвера — стабильный плюс.

## ПЛАН И СТАТУС
- [x] Шаг 1: NLHE GTO push/fold солвер (pushfold_solver) — exploitability~0, диапазоны как Nash-чарты.
- [x] Шаг 2: LBR-измеритель эксплуатируемости агента (lbr_eval) — даёт нижнюю границу в bb/100.
- [x] Шаг 3: постфлоп-CFR (river_solver) — exploitability->~0, стратегия поляризована (барбель), реальные руки.
- [~] Шаг 4: 
    - [x] 4a: turn-солвер (turn_solver) — CFR + chance-слой ривера, сходится, поляризация.
    - [x] 4b: интеграция GTO push/fold в агента (pushfold_gto.hpp + arena_v2 --stackbb): на коротком
          HU-префлопе агент играет ТОЧНУЮ GTO push/fold (10bb vs tight +6 bb/100, vs station +59).
    - [x] 4c: in-play re-solving (resolve) — агент решает поддерево (turn/river) на лету,
          выдаёт GTO-действие для своей руки (сет→бет 1.0, топ-пара→чек, мусор→чек/блеф). Проверено.
    - [x] 4d: ядро префлоп-CFR (blueprint) построено и ЧЕСТНО показывает предел: без реального
          постфлоп-EV стратегия вырождается (пере-шов). Корректная короткостековая префлоп-GTO =
          pushfold_solver (Nash). Полный deep-stack preflop+flop blueprint = компьютер-bound (масштаб солвера).

## Что РЕАЛИЗОВАНО и ПРОВЕРЕНО

1. GTO-движок (CFR) — `tools/cfr_solver.cpp` → `bin/cfr_solver`
   Counterfactual Regret Minimization, ДОКАЗУЕМО сходится к точному равновесию:
   на Kuhn poker exploitability(NashConv) → ~0.001, стратегия совпадает с
   аналитическим GTO (K бетит 3x от блефа J, Q чек). Это корректное ядро
   равновесия; тот же алгоритм масштабируется на Leduc/NLHE.
       ./bin/cfr_solver --iters 200000

2. Прайор Pθ (имитация GTO на PokerBench) — `train_v2` + `poker_v2.hpp`
   MLP на 80 фичах (рука×борд, дро, pot-odds), 8 классов сайзинга. val_acc ~0.75.
   По диагностике (`probe_v2`) понимает цену (P(fold) растёт с ценой) и вэлью
   (поляризованный барбель). Трансформер-вариант — опционально (USE_TF_PRIOR=1),
   но на seq_len=1 он уступает MLP.

3. Онлайн-агент BAEM — `arena_v2.cpp` (место `model2x`)
   Прайор + оценка σ̂_opp по активным соперникам + λ̂=1−D_KL/Dmax + α*(t) с гейтингом
   h(ε)=1−C/√t + EV-эксплойт (мультивей-эквити, контроль дисперсии) +
   π=(1−α*)πGTO+α*softmax(EV/τ). A/B на реальном PokerBench: фикс-прайор +475 BB/100,
   полный BAEM +757 (онлайн-слой даёт +60%, стейшна доедает вдвое).

4. Оценка и диагностика — `arena_v2` (турнир/A-B), `probe_v2` (канонические споты,
   пот-оддсы, агрессия vs эквити).

## Что ОСТАЁТСЯ (большая инженерия, честно)

A. Масштаб CFR на полноценный NLHE: абстракция рук (бакеты) + дерево сайзингов +
   MCCFR/Deep CFR, миллионы итераций, тяжёлый компьютинг (недели). Ядро (cfr_solver,
   gto_engine/cfr_engine.hpp) — фундамент для этого.
B. Depth-limited re-solving в момент игры (ReBeL/DeepStack) — главный скачок к
   солвер-силе; поверх CFR-стратегии.
C. Корректный NLHE best-response (LBR) для замера эксплуатируемости самого агента
   в mbb/g (info-set BR, как в cfr_solver, но на дереве NLHE).

## Запуск

    bash tools/run_v2.sh                      # обучить прайор + турнир
